## 3. P0: 바로 고쳐야 할 사용성 문제

### P0-1. 업무 보드 카드에서 버튼이 콘텐츠를 밀어내고 있음

**증거**
- `05-daily-board-all.png`, `06-daily-board-today.png`
- `DailyBoardWindow.xaml:29-33`에서 카드 상단 `DockPanel` 오른쪽에 `기한 설정`, `완료`, `열기`, `더보기` 4개 버튼이 항상 배치된다.
- `DailyBoardWindow.xaml.cs:459`에서 제목을 `CompactLine(task.Title, 44)`로 한 번 잘라낸 뒤, XAML에서도 `TextTrimming`이 적용되어 실제 UI에서 제목 복구가 불가능하다.
- 메인 업무 목록도 `MainWindow.xaml:127-130`에서 `기한 변경`, `열기`, `숨김`을 오른쪽에 두고, `MainWindow.xaml.cs:1381`에서 제목을 `CompactLine(task.Title, 54)`로 잘라낸다.

**사용자 체감**
- 사용자는 메일 기반 업무 앱에서 “무슨 업무인지”를 먼저 봐야 하는데, 지금은 버튼이 먼저 보인다.
- 카드가 많은 날에는 같은 버튼 4개가 반복되어 전체 화면이 버튼 표처럼 보인다.
- 사용자가 이미 말한 것처럼, 버튼이 오른쪽에 있어서 내용이 잘 안 보인다.

**권장 방향**
1. 카드 첫 줄은 제목 전용으로 확보한다. 최소 1줄, 가능하면 2줄 wrap.
2. 날짜/발신자/수신일 meta row 아래에 작은 action row를 둔다.
3. 기본 노출 액션은 최대 2개만 둔다: `완료`, `열기`.
4. `기한 설정/변경`, `나중에 보기`, `숨김`은 `더보기`로 보낸다. 단, 기한이 없는 카드만 작은 `기한 추가` 링크/칩을 노출할 수 있다.
5. 카드 전체 영역 더블클릭/Enter는 `열기` 또는 상세 열기로 연결한다.

**추천 카드 구조**

```text
[할 일] 오늘 회신 · 결제 플로우 문구 확인
D-day · Design Partner · 05-16 08:40
[완료] [메일 열기]                         [⋯]
```

또는 사용자가 제안한 방식처럼 버튼을 날짜 위치 아래로 보내면, 제목 영역이 확보되어 업무 파악 속도가 훨씬 좋아진다.

### P0-2. `최근 1개월 스캔`은 버튼명으로 부적절함

**증거**
- `MainWindow.xaml:74` 버튼 라벨이 `최근 1개월 스캔`.
- 실제 스캔 기간은 설정값 `RecentScanDays`에 의해 바뀌며, 스캔 후 상태는 `최근 30일 메일 12건 확인...`으로 표시된다.
- 설정에도 `최근 스캔 기간(일)`이 있어 버튼과 설정이 느슨하게 연결된다.

**문제**
- 사무업무 사용자는 “스캔”이라는 단어보다 “지금 새 메일 확인”이 더 자연스럽다.
- “최근 1개월”이 버튼에 박혀 있으면 변경 가능한 설정처럼 느껴지지 않는다.
- 업무 보드의 `30일 내` 필터와 혼동된다. 하나는 메일 읽기 범위, 하나는 업무 기한 필터인데 둘 다 기간처럼 보인다.

**권장 문구**
- Primary CTA: `지금 메일 확인`
- 보조 상태: `최근 30일 기준 · 마지막 확인 21:41`
- 설정명: `메일 확인 범위: 최근 30일`
- 업무 보드 필터: `오늘`, `이번 주`, `이번 달`, `기한 없음`, `전체`처럼 due 기준임을 더 명확히 한다.

### P0-3. 내부/개발자 문구가 사용자 화면에 노출됨

**증거**
- `MainWindow.xaml:94`: `설정은 탭 안에 정리했습니다`
- `MainWindow.xaml:233`: `자동 watcher 요청(운영 smoke gate 후 활성)`
- `FixtureMailSource.cs:153`: `Fixture 메일을 열었습니다.`
- `MainWindow.xaml.cs:751`: fixture source이면 `Fixture 원본 메일을 여는 중입니다…`

**문제**
- `설정은 탭 안에 정리했습니다`는 사용자가 알아야 할 현재 상태나 다음 행동이 아니다. 릴리즈 노트/개발 메모처럼 보인다.
- `watcher`, `smoke gate`는 QA/개발 용어다. “새 메일 자동 확인”이 켜졌는지, 언제 확인하는지가 사용자에게 중요하다.
- `Fixture 메일`은 QA 프로필에서는 이해 가능하지만, 사용자 화면에서 보이면 신뢰가 깨진다. 샘플/데모 모드라는 맥락 없이 `Fixture`가 나오면 오류처럼 보인다.

**권장 치환**
- `설정은 탭 안에 정리했습니다` → 제거. 대신 필요하면 `자동 확인 꺼짐`, `마지막 확인 21:41`, `검토 후보 3개` 같은 상태 badge 사용.
- `자동 watcher 요청(운영 smoke gate 후 활성)` → `새 메일 자동 확인` + 보조문구 `수동 확인이 한 번 성공하면 15분마다 확인합니다.`
- `smoke gate` → `안전 확인` 또는 UI에서는 숨김.
- `Fixture 메일을 열었습니다.` → `샘플 메일입니다. 실제 Outlook 원본은 없습니다.` 또는 샘플 프로필에서는 `열기` 버튼 비활성/`샘플 보기`로 변경.

### P0-4. “실시간으로 메일이 올 때마다 확인”은 현재 구현/설정상 보장되지 않음

**코드 근거**
- 앱 로드 시 자동 스캔은 `_settings.AutomaticWatcherRequested && _settings.SmokeGatePassed`일 때만 수행됨: `MainWindow.xaml.cs:344-348`
- 자동 확인 timer는 15분 간격 polling: `MainWindow.xaml.cs:527-540`
- 수동 스캔 성공 후에만 smoke gate 통과 처리: `MainWindow.xaml.cs:1062-1075`
- 기본 설정은 `AutomaticWatcherRequested: false`, `SmokeGatePassed: false`: `RuntimeSettings.cs:28-45`
- 현재 앱 내 진단 결과(`artifacts/visual-qa/app-diagnostics.json`)는 `AutomaticWatcherEnabled: false`, `Mode: manual`, reason `outlook-com-unavailable`, `outlook-inbox-unreadable`, `automatic-watcher-not-requested`.

**진단**
- 지금 구현은 “새 Outlook 메일 도착 이벤트마다 즉시 반응”이 아니라 “조건 충족 후 15분마다 보수적 read-only 스캔”이다.
- 따라서 사용자가 기대하는 “실시간”과 현재 제품 동작 사이에 간극이 있다.
- 현재 테스트 환경에서는 Outlook COM ProgID 자체가 없어 실제 Outlook 연동 확인은 실패했다. 이는 앱의 진단이 잘 잡아낸 것이지만, UI에서 이 상태를 더 알기 쉽게 보여줘야 한다.

**권장 UX**
- 헤더 또는 설정에 명확한 상태 표시:
  - `자동 확인 꺼짐`
  - `자동 확인 준비 필요: Outlook 연결 확인 필요`
  - `15분마다 확인 중 · 마지막 확인 21:41 · 다음 확인 21:56`
- “실시간”이라는 말을 쓰려면 실제 Outlook event subscription 또는 더 짧은 polling/백오프 정책이 필요하다.
- 당장은 “실시간” 대신 `주기적 자동 확인`이라고 표현하는 것이 정직하다.

## 4. P1: 정보 구조/레이아웃 개선

### P1-1. 메인 화면과 업무 보드 창의 역할이 중복됨

메인 안에도 `업무 보드` 탭이 있고, 상단 버튼으로 별도 `업무 보드` 창도 열린다. 두 화면은 비슷한 목록을 보여주지만 레이아웃/액션/필터가 다르다. 사용자는 어떤 보드가 진짜인지 헷갈릴 수 있다.

**권장**
- 하나를 primary로 정한다.
  - 추천: 별도 업무 보드 창을 primary 업무 처리 화면으로 두고, 메인은 상태/설정/검토 후보/빠른 추가 중심으로 축소.
  - 또는 메인 창 자체를 업무 보드로 만들고 별도 창은 제거.
- 버튼명도 `업무 보드` 중복보다 `전체 보드 열기`, `오늘 보드 열기`처럼 진입 목적을 명확히 한다.

### P1-2. `내가 할 일` / `기다리는 중`은 유지 가치가 있지만, 카드 안의 `할 일`/`일정` 반복은 줄일 필요가 있음

제품 정의상 “내가 해야 할 일”과 “내가 요청하고 기다리는 일”의 분리는 매우 유용하다. 사무업무 사용자에게 실제로 필요한 구분이다. 다만 카드마다 큰 badge로 `할 일`, `일정`, `대기`를 반복하면 제목 공간이 줄어든다.

**권장**
- 보드 컬럼 `내가 할 일` / `기다리는 중`은 유지.
- 카드의 `할 일`, `일정` badge는 더 작게 만들거나 제목 뒤 작은 chip으로 이동.
- `일정`은 일정성 업무일 때만 캘린더 아이콘/작은 tag로 표현.
- main tab의 `할 일/일정` 같은 별도 좌측 분류가 필요하다면 필터로 두되, 기본은 전체 업무 흐름을 보여주는 것이 좋다.

### P1-3. TabControl 기본 스타일이 전체 카드 디자인과 충돌함

현재 탭 헤더가 Windows 기본 테두리/회색 선으로 보이고, 둥근 카드 UI와 시각 언어가 다르다. 전체적으로 “디자인 시스템을 입히다 만 느낌”을 준다.

**권장**
- 기본 `TabControl` 대신 segmented control/pill navigation으로 통일.
- 탭 영역의 큰 회색 border 제거.
- 현재 화면 제목(`오늘 업무`)과 선택 탭 상태가 한 덩어리로 읽히게 구성.

### P1-4. 검토 후보 탭은 “왜 검토해야 하는지”가 부족함

검토 후보는 낮은 확신 후보를 안전하게 분리하는 중요한 장치다. 하지만 현재 카드에는 후보 제목만 보이고, confidence/근거/발신자/수신일/마감이 충분히 드러나지 않는다.

**권장 카드 구조**

```text
검토 후보 · 전달 메일 담당자 확인
내 업무인지 불명확 · Account Owner · D-1 05-17 09:00
근거: 전달 메일이라 담당자 확인 필요
[업무로 등록] [원본 열기] [나중에] [무시]
```

버튼은 첫 번째 선택 항목이 있을 때만 상단에 크게 두고, 카드별 액션은 compact하게 둔다.

### P1-5. 설정 화면은 비개발자에게 부담이 큼

설정 상단부터 LLM endpoint, 모델, 고급 watcher, timeout, token env var 등이 보여서 “업무 비서 앱”보다 “개발자 도구”처럼 느껴진다.

**권장 IA**
1. 일반
   - Windows 시작 시 실행
   - 메일 확인 범위
   - 새 메일 자동 확인 on/off
2. 알림
   - 오늘 브리핑 시간
   - 알림 미리 보기
3. AI 분석(고급)
   - LLM 사용 여부
   - endpoint/model/test
4. 문제 해결
   - 진단 실행
   - 알림 테스트
   - 업무 데이터 초기화
   - JSON 상세보기는 접기/복사 버튼

## 5. P2: polish / 신뢰감 개선

### P2-1. 버튼 크기와 위계

현재 버튼이 모두 큼직하고 반복되어 primary/secondary가 흐려진다. 업무 카드에서는 작은 버튼/아이콘/overflow가 더 적합하다.

- Primary: `지금 메일 확인`, `완료`
- Secondary: `메일 열기`
- Tertiary/overflow: `기한 변경`, `나중에 보기`, `숨김`

### P2-2. 날짜/기한 위치

사용자가 제안한 “버튼을 날짜 위치 밑으로”는 좋은 방향이다. 다만 날짜 자체도 제목 바로 밑에 있어야 업무 우선순위 판단에 도움이 된다. 추천은 다음 순서다.

1. 제목
2. D-day/발신자/수신시간
3. 액션 버튼 row

### P2-3. 빈 상태/완료 상태

업무가 없을 때 `표시할 업무가 없습니다`만으로는 다음 행동이 약하다.

- `표시할 업무가 없습니다. 지금 메일 확인을 눌러 최근 메일에서 후속조치를 찾아보세요.`
- 자동 확인 꺼짐이면 `자동 확인이 꺼져 있습니다.` CTA 제공.

### P2-4. 진단 결과

JSON 자체는 개발/지원에는 좋지만, 일반 사용자에게는 먼저 요약이 필요하다.

예:

```text
Outlook 연결: 실패 — Classic Outlook을 찾지 못했습니다.
자동 확인: 꺼짐 — 설정에서 켤 수 있습니다.
로컬 저장소: 정상
알림: 정상
LLM: 꺼짐
[상세 JSON 복사]
```

## 6. 권장 개편안

### 6-1. 메인 헤더

현재:

```text
[업무 보드] [최근 1개월 스캔]
```

추천:

```text
MailWhere
놓치기 쉬운 메일 후속조치를 모아둡니다.
자동 확인 꺼짐 · 마지막 확인 21:41
[지금 메일 확인] [오늘 보드]
```

`설정은 탭 안에 정리했습니다` 배지는 제거.

### 6-2. 업무 보드 카드

현재:

```text
[할 일] 제목...        [기한 설정] [완료] [열기] [더보기]
D-day | sender | received
```

추천:

```text
[할 일] 오늘 회신 · 결제 플로우 문구 확인
D-day · Design Partner · 05-16 08:40
[완료] [메일 열기]                              [⋯]
```

기한 설정은 다음 중 하나:
- 기한이 없는 카드만 `기한 추가` 표시
- 기한 있는 카드에서는 날짜 chip 클릭으로 변경
- 모든 기한 관련 액션은 `⋯` 아래로 이동

### 6-3. 자동 확인 상태

설정의 `자동 watcher 요청(운영 smoke gate 후 활성)` 대신:

```text
새 메일 자동 확인 [끔/켬]
수동 확인이 한 번 성공하면 15분마다 새 메일을 확인합니다.
마지막 확인: 21:41 · 다음 확인: 21:56
```

Outlook 연결 실패 시:

```text
Outlook 연결이 필요합니다. Classic Outlook을 실행한 뒤 진단을 다시 실행하세요.
```

### 6-4. 검토 후보

검토 후보는 업무 보드에 섞지 않는 현재 방향이 맞다. 다만 “왜 후보인지”를 보강해야 한다.

- 후보 사유 badge: `담당자 불명확`, `마감 불명확`, `LLM 실패`, `전달 메일`
- Confidence는 숫자보다 사용자 언어: `확신 낮음`, `확인 필요`
- Bulk 처리: 여러 후보 선택 후 등록/무시

## 7. 현재 Outlook/실시간 확인 진단

### 확인 결과

현재 직접 실행한 앱 진단 결과:

```json
{
  "AutomaticWatcherEnabled": false,
  "Mode": "manual",
  "ReasonCodes": [
    "outlook-com-unavailable",
    "outlook-inbox-unreadable",
    "automatic-watcher-not-requested"
  ]
}
```

이 결과는 현재 실행 환경에서 Classic Outlook COM을 사용할 수 없고, 자동 확인도 요청되지 않았다는 뜻이다.

### 구현상 동작

- 기본값은 자동 확인 꺼짐.
- 사용자가 자동 확인을 요청해도, 수동 스캔이 한 번 성공해 smoke gate가 통과해야 한다.
- 통과 후에는 15분마다 `ScanRecentMailAsync(showSummaryNotification: false)`를 실행한다.
- 즉, “메일이 오는 즉시”가 아니라 “15분마다 최근 기간을 다시 읽어 반영”이다.

### 제품적으로 필요한 표현

사용자가 “Outlook 메일이 실시간으로 올 때마다 확인되는지”를 궁금해하는 것은 자연스럽다. 현재 UI는 이 질문에 답하지 못한다. 최소한 아래 상태가 보여야 한다.

- 자동 확인: 켜짐/꺼짐
- 확인 방식: 15분마다 / Outlook 이벤트 기반
- 마지막 확인 시간
- 다음 확인 예정 시간
- Outlook 연결 상태
- 실패 시 이유와 복구 방법

## 8. 우선순위별 실행 제안

### 1차 수정(P0, 가장 효과 큼)

1. 업무 카드 액션 버튼을 오른쪽에서 아래 row/overflow로 이동.
2. card title의 코드 레벨 `CompactLine` 제거 또는 대폭 완화. UI가 wrap/trim을 결정하도록 변경.
3. `최근 1개월 스캔` → `지금 메일 확인`으로 변경.
4. `설정은 탭 안에 정리했습니다` 제거.
5. `자동 watcher`, `smoke gate`, `Fixture` 문구를 사용자 언어로 치환.
6. 헤더에 자동 확인 상태/마지막 확인 표시 추가.

### 2차 수정(P1)

1. 메인 업무 탭과 별도 업무 보드 창의 역할 정리.
2. 기본 TabControl 스타일을 custom segmented nav로 변경.
3. 설정 화면을 일반/알림/AI/문제 해결로 재구성.
4. 검토 후보 카드에 후보 사유/날짜/발신자/근거 표시.

### 3차 수정(P2)

1. 알림 히스토리 또는 최근 알림 영역 추가.
2. Empty state CTA 개선.
3. 실제 Outlook event subscription 검토 또는 polling UX 명확화.
4. 카드별 keyboard shortcut/focus ring 정리.

## 9. 결론

현재 MailWhere는 기능적으로 “메일 후속조치 보드” 방향은 맞다. 그러나 시각적으로는 사무 사용자가 매일 켜두고 믿기에는 아직 개발자/QA 흔적과 버튼 과밀이 많이 보인다. 가장 먼저 고칠 것은 디자인 색상보다도 **업무 카드 정보 우선순위**다. 제목과 마감/발신자를 먼저 읽히게 하고, 반복 액션은 작게 접으면 사용성이 바로 개선된다. 그 다음으로 `지금 메일 확인`, `자동 확인 상태`, `마지막/다음 확인 시간`을 명확히 보여주면 “이 앱이 내 메일을 제대로 지켜보고 있나?”라는 핵심 불안을 줄일 수 있다.
